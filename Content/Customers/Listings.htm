Fields and options defined

After you've chosen the listing you want to manage from the drop-down menu, the page opens. The functionality appears the same on most pages. The values of each column are defined below. If you do not make any selections for a listing type, no options will be presented to your customers for these listing types.

If you use your logo within the templates, the logo.gif must match your actual logo.

After you make any changes or edits on each line, click the Save button at the bottom of the page to save your changes.

         Fraud Codes (AVS)

    The AVSAddress Verification System can be your first line of defense against fraudulent transactions. Fraud codes are configurable based upon the payment processor that is selected by your customers. The settings displayed at Commerce > Settings > FraudLogic > AVS Settings are pulled from the settings on this page.

    Customized settings can be configured for these credit cards:

    • Fraud codes - American Express
    • Fraud codes - Discover
    • Fraud codes - MasterCard
    • Fraud codes - Visa

    To make the most of this system it is important to understand how it works. AVS attempts to match the billing address entered as part of ordering process against the address listed on the credit card statement, as recorded

    by payment processors. To overcome the problems with abbreviations, misspelling, etc... the AVS strips away all letters and attempts to match only the numbers in the street addresses and the zip code. No street names or the name of the state are checked during this process.

    The AVS information is only obtained back from the processor after an authorization is performed. If the merchant wishes to reject those transactions which fail to meet the desired AVS matching level then the system will perform an immediate void of the transaction, inform the customer that their address did not match and allow them to enter a new address. Since an authorization has already occurred, most merchant processors will assess their per transaction fee for this authorization even though the transaction was voided.

    Go to Commerce > Settings > FraudLogic > AVS Settings to learn more.

    Newsletter Signup Option

    The MailChimp integration allows for customer data from newsletter signups to be sent to MailChimp. A master page is available at Content > Pages > Masters > Page Sections > Newsletter Signup Option. When the master is populated, a checkbox displays with custom text. When the checkbox is checked, the customer information is automatically sent out to MailChimp.

    In order for the newsletter signup option to function, the Toro job must have a value in the 'customfield2' that contains the MailChimp List ID. Otherwise, the emails in the reserve table won't be exported.

    EOM sends this information to MailChimp. The frequency in which the data is sent to MailChimp, is handled with the Job Scheduler.

         Return Types
    Site Manager Pages:
    CRM > Communications > Listings
    Commerce > Orders > View > Options > RMA Request
    Content > Pages > Masters

    Knowledgebase Article #1138: Click to learn more.

    This page is accessible when an RMA is requested in the Site Manager. To see this menu, go to Commerce > Orders > View and open an order.

    The Return Types listing also displays to your customers in the "Return Reason" in the Members Area of your site. To request a return, a customer needs to login to their account and click on My Order History. They can then click to request a return for a particular order.

    This functionality enables you to edit the content in the Select Return Reasons drop- down menu on the "RMA Request" pop-up window. You can set up all applicable return types here.

    • Display Text - As a reminder, this is the text your customers will see on your site.
    • Lookup Value - This should be set with an internal value that works with your API. For some return reasons, you may want to have the customer call to coordinate the return rather than allowing the customer to generate the RMA via your website. Because of this, you can prevent an RMA from being requested by a customer from the Member Area on your site. The platform allows you to control which return reasons are applicable to online RMA generation based on the return reason selected by a customer.
      • Preventing Online Returns. If you want the return reason to prevent an on-line RMA from being generated, enter the value of -99 in the Lookup Value field. If a customer selects a return reason that you have set with a value of -99, they won't be allowed to submit an RMA. You can edit the text that will display on your site in the master in Content > Pages > Masters > RMA Not Allowed. Once created, the text will appear in the pop-up window indicating the reason why the customer can't generate an RMA for this type of return.

    • Template - You can edit email templates directly within each Return Type you have defined. This gives you more flexibility to create emails specific for each return reason. Click on the template icon to edit the template for each individual line item. For example, if a customer indicates they are returning the merchandise because it does not fit, you can have a customized email sent for this reason code. As a reminder, the email templates that are customized for specific return types will override the default email for Return Types.

    • Default Record - You must define the Return Types that will allow shipping labels to be requested and produced. In the Default Record column, place a checkmark in each return reason in which you want to enable the system to generate a shipping label.

    Note: If you want to be able to pull return requests via your API, use the Aptos RMA API. Once you have set up the API, test it to make certain it is working properly. https://api.Aptos.com/wiki/Order%20-%20GetOrdersForRMA.ashx

         Wishlist Events
    Site Manager Page: Users > Website Settings > Site Layout > Website Settings
    Knowledgebase Article #2111: Click here to learn more.

    In the Wishlist Events area you can create a variety of types of wishlists from which your customers can choose. Then, when your customer wants to create a wishlist, they can choose from the options you create here. Examples of wishlists are: weddings, birthdays, anniversaries, and so on.

    If you want to allow wishlists to be used on your site, set the wishlist master settings at Users > Website Settings > Site Layout > Website Settings.

    • Display Text - This text will be displayed in the members area of your site in a drop-down menu so your customers can select the type of wishlist they are creating if the Default Record is selected.
    • Lookup Value - This text will be displayed in the members area of your site in a drop-down menu so your customers can select the type of wishlist they are creating if the Default Record is selected.
    • Option Value - Not required.
    • Default Record - Select the box so that the drop-down menu will be populated. The default record is tied to shipping information. So, for example, after an event is over, the items can be sent to a different shipping address.

    These settings enable your customers to create wishlists and they can choose to make their wishlist public. Visitors to your site can search for customers and find their wishlists. This is useful for example, in the case of a wedding registry. In conjunction with this feature, when the registrant indicates the wishlist is a gift registry, all contact information displays and the shipping information is populated. In addition, co-registrants can be added to the wishlist. This is useful in the case of a wedding registry. If a registrant updates their wishlist, an email is sent to the co-registrant notifying them of the changes.
