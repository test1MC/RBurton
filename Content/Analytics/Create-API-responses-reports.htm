Creating API Responses reports

The APIapplication programming interface Responses report captures all information pertaining to each request made through the API. This allows Aptos to more easily troubleshoot issues as they arise as well as to better monitor the overall performance of the API.

You can filter the report based on the different API methods being used, and the report shows those API methods used rather than all of the available API responses.

To create an API Responses report

  1. In Analytics, select User Reports > API Responses.
  3. Optionally, choose an API method from the API Method list.
  4. Click Search.
  5. Click on any report table column header to sort information in that column: IP Address, Submit and Response Times, the API Service/Method used, and the Response.

Using the API Responses report

The report contains three graphs and as a table complete with summary information

About report graphs

The report includes three graphs:

  • a pie chart on the left shows each request made to the API for the specified time frame. It is divided into sections representing percentages of the actual API method called.
  • a pie chart on the right shows the percentage breakdown of errors compared with successful responses.
  • a graph below the charts displays a line for each method called, along with the number of seconds that each request takes. This shows how frequently the API methods are used, and helps to identify performance issues that may exist.

About the report table

A table displayed below the graphs shows the start and end times of each request, the elapsed time of the request, whether it was successful, and what method was actually called. Click the Details icon to see the request made along with its corresponding response.

The detailed response of the API helps your partners or technologists to determine what might be wrong with their request. For example, if the XML is malformed or cased incorrectly, this will be easy to see. This also helps Aptos support staff to replicate each request in the event that there is a problem with the API itself.
